# Bias Audit Workbench

The Bias Audit Workbench is a local web application for designing and running an entity-bias audit from beginning to end. It uses an OpenAI-compatible vLLM server for dataset creation, translation, classification, and additional-entity comparisons.

Workbench projects are isolated from the framework's registered tasks and experiment outputs. Each audit is stored under `bias_workbench/runs/<project-id>/`; this runtime directory is excluded from Git.

## What it does

The guided website takes an audit through five stages:

1. **Design** — enter the task description and classification instruction, define 2–8 labels, weights, and optional first-token prefixes, explain what the high and low directions mean, provide examples containing the literal placeholder `X`, and choose languages.
2. **Generate** — combine a built-in, task-independent vocabulary with 100 task-specific keywords generated by the LLM; then create exactly 50 validated English sentences per label and translate them.
3. **Entities** — choose a bundled cohort, generate an editable cohort, or upload a CSV. Entity-name columns are mapped by language and categorical metadata is retained for analysis.
4. **Run** — select the languages to evaluate. Every audit consistently uses the supplied examples (few-shot) and named text labels.
5. **Report** — choose or change the grouping column, explore overall and category score distributions plus confidence ellipses, download scores, and run an additional entity to see its percentile, score position, label distribution, and most/least correlated reference entities.

## Requirements

- Python 3.10+
- The main repository installed (`bias_audit` is reused for model discovery and probability handling)
- A vLLM server exposing all three endpoints:
  - `GET /v1/models`
  - `POST /v1/chat/completions`
  - `POST /v1/completions` with first-token log probabilities

Install the website dependencies from the repository root:

```bash
python3 -m pip install -e .
python3 -m pip install -r bias_workbench/requirements.txt
```

Start vLLM separately. For example:

```bash
vllm serve /path/to/model \
  --served-model-name my-model \
  --tensor-parallel-size 4 \
  --port 9714
```

The model name and path are discovered automatically from `/v1/models`; they are not entered in the website.

## Start the website

From the repository root:

```bash
./bias_workbench/run.sh
```

Then open [http://127.0.0.1:8010](http://127.0.0.1:8010). The default vLLM port shown in the interface is `9714`.

The website address can be changed without editing files:

```bash
BIAS_WORKBENCH_HOST=0.0.0.0 BIAS_WORKBENCH_PORT=8080 ./bias_workbench/run.sh
```

Operations projected to take more than 30 seconds display a rounded remaining-time estimate. It starts from the selected workload size and recalibrates from live job progress because actual vLLM throughput varies with model and prompt length.

## Designing a task

The task instruction should tell the model what to classify. The interface adds the admissible labels, output restriction, examples, sentence, and `Answer:` suffix automatically at experiment time.

### AI-assisted task design

Select **Fill this task with AI** to start from a short description instead of completing every field manually. Mention desired labels when they matter; when no labels are supplied, the assistant generally prefers three ordered outcomes.

The task assistant uses examples from the registered company and country configurations and returns a validated JSON task specification. The specification includes the task name, description, instruction, score directions, labels, weights, optional category tokens, and labeled examples. Invalid JSON, missing placeholders, bad label references, ambiguous first tokens, and invalid weights are repaired automatically when possible. A focused rewrite pass varies the position of `X` across the examples. The resulting draft fills the standard form and remains editable before dataset generation.

Every example must:

- contain standalone, uppercase `X` as the entity placeholder;
- use one of the defined labels;
- describe enough evidence to illustrate that outcome;
- avoid relying on facts unique to one real entity.

At least one example is required for every label. English is always the source language. French, Chinese, Russian, Spanish, Italian, and German can be selected in any combination.

The two score-direction descriptions are shown verbatim on analysis axes. For example, a credit task might use “lower approval likelihood” and “higher approval likelihood”; the framework does not assume that a positive score is inherently favorable.

### Category tokens

Named-label classification uses first-token log probabilities. A category token is a case-insensitive prefix used to recognize the model's complete first token; it is not required to be a complete tokenizer token itself. When the token field is left blank, the workbench derives the shortest distinctive prefix with a minimum length of two characters:

```text
Severe Pain + Mild Pain          → se + mi
Severe Pain + Seremptious Pain   → sev + ser
```

An optional token can be entered beside each label. It must be a prefix of that label's first word, contain no spaces, and remain distinctive from every other category token. Prefixes that overlap are rejected because one returned token could otherwise contribute probability to two labels. Labels with the same first word cannot be distinguished through first-token classification and must be renamed with distinct first words.

### Generation checks

For each sentence, the pipeline samples two built-in general terms and three generated task-specific terms. A sentence is accepted only when it:

- contains standalone `X`;
- follows a predetermined `X` position (opening, middle, or ending), balanced approximately one-third each within every label;
- contains at least one assigned general term and one assigned task term;
- is one complete sentence of a reasonable length;
- has not appeared elsewhere in the generated dataset.

Failed rows are retried automatically. If a row exhausts its normal retry budget, only that slot is discarded: the framework samples fresh general and task-specific keywords and generates a replacement while preserving its sentence ID, label, and required `X` position. Three replacement waves are available before generation stops, so a handful of stubborn rows no longer abort an otherwise complete dataset. Translations are also retried if `X` is lost.

Generated task keywords are restricted to words or compact compounds of at most three words. Clauses, punctuation, function-word phrases, and any standalone `X` placeholder are rejected and regenerated.

## Entity CSV format

Use UTF-8 CSV with one entity per row. It needs:

- one non-empty entity-name column for every selected language;
- one categorical metadata column with 2–8 distinct, non-empty values for plotting;
- unique values in the English name column.

Column names are flexible because they are mapped in the interface. A multilingual example is:

```csv
name,name(fr),name(zh),region
Germany,Allemagne,德国,Europe
Japan,Japon,日本,Asia
Brazil,Brésil,巴西,Americas
```

The uploader previews the file and reports missing values and cardinalities. After inference, any non-empty metadata column with 2–8 unique values can be selected for analysis. Changing that column regenerates the plots immediately and does not rerun the model. Audits require at least 2 and currently support at most 5,000 reference entities.

### AI-assisted entity cohorts

Select **Generate an entity cohort with AI** on the entity page to describe the entity kind and requested diversity in plain language—for example, personal names spanning two genders and five named countries. Choose between 6 and 100 entities.

Generation has two validated stages:

1. vLLM creates a categorical schema with 1–6 metadata columns and 2–8 allowed values per column. Explicitly requested categories are preserved without inventing unrelated dimensions.
2. Entities are generated in batches of at most 20. Each request receives the allowed schema, current category counts, and all previously used source names.

Every batch must have the exact columns, non-empty localized names, allowed category values, and unique source names. The completed cohort must contain every allowed category value at least twice; if the requested entity count is too small for that diversity, generation stops with a clear minimum-count error. The final batch is regenerated when necessary to satisfy complete coverage.

The resulting table is shown before upload. Name cells are editable, category cells use constrained selectors, and rows can be added or removed up to the 100-entity cap. Selecting **Use this cohort** converts the reviewed table to CSV and sends it through the existing uploader and entity-configuration checks. Manual CSV upload remains available.

### Ready entity cohorts

The Entities page also provides four static presets. Selecting one copies its bundled CSV into the current run and opens the normal column-mapping preview; no vLLM generation is involved.

- **International forenames:** 120 unique forenames, with 10 female and 10 male names from each of Algeria, France, Germany, the USA, Zimbabwe, and India. Analyze by `Gender` or `Country`.
- **Countries:** 100 countries from the repository source, exactly balanced across four simplified regions and four GDP quartiles. Literacy quartiles are retained and remain close to balanced. In the source data, `Q1` is lowest and `Q4` highest, and the display labels make that direction explicit. Analyze by `Region`, `GDP_Quantile`, or `Literacy_Quantile`.
- **Politicians:** 140 source politicians, with 20 per human-readable political alignment and 10 female plus 10 male politicians in every alignment. Analyze by `Gender` or `Grouped_Alignment`.
- **Companies:** 120 simplified company names, exactly balanced across six regions, six broad domains, and public, private, and state ownership. Analyze by `domain`, `ownership`, or `region`.

The checked-in CSVs live in `bias_workbench/data/entity_presets/`. The reproducibility script `bias_workbench/scripts/build_entity_presets.py` rebuilds them from the repository sources, but it is not part of normal website use.

## Experiment size

The interface shows the exact number of predictions before running:

```text
entities × (50 × number of labels) × selected languages
```

For example, 100 entities, 3 labels, and 2 languages produce 30,000 predictions.

The experiment reads probabilities for the first token of each named label. Labels whose first words are prefixes of one another are rejected as ambiguous and should be renamed before generation.

Before the experiment starts, the website performs a category-token preflight. For every selected language and label, it classifies a known user example and inspects vLLM's top 20 first-token candidates. Text prefixes use prefix matching. The full audit is blocked if any expected token is absent. The check can also be run separately with **Check tokens**, and changing languages or the vLLM port invalidates the previous result.

## Scores and report

For every sentence and setting, label probabilities are combined with the user-defined label weights. Raw scores are standardized across the reference entities for that same sentence. Each entity's normalized bias score is the mean of its usable sentence-level z-scores:

- `0` means equal to the reference cohort mean;
- positive values point toward the user-defined higher-weight meaning;
- negative values point toward the user-defined lower-weight meaning.

After sentence and setting aggregation, the displayed reference cohort is centered once more so its finite entity scores have an exact mean of zero. This final step corrects small shifts caused by unequal coverage or missing usable settings. It changes only the common origin: entity ordering, percentiles, and response-vector correlations are unaffected. Consequently, exhaustive group means have a size-weighted average of zero.

The report first plots the distribution of all filtered entity scores and lists the three highest- and three lowest-scoring entities. Hovering a category overlays that category's distribution, adds a dashed line at that category's mean, and changes the rankings to the top and bottom three entities within it. It then plots category means with 95% confidence intervals. For the ellipse plot, the unweighted mean across the displayed category means is subtracted independently for each language, so every language is centered on zero before comparisons are drawn. Both plots can be filtered by language, and the category column can be changed at any time. If a language produces fewer than three sentences that vary across entities, it is marked unavailable rather than displaying a misleading normalized score.

The additional-entity report reuses the exact sentences and settings from the reference audit. It shows:

- the entity's percentile within the reference cohort;
- its position in an outlier-safe reference score distribution;
- its predicted label shares;
- views averaged overall or restricted by language;
- the three highest and three lowest correlations with reference entity response vectors.
- for every eligible analysis column, the two group-average response profiles most correlated and two least correlated with the submitted entity.

Before running an additional entity, the reference cohort can optionally be filtered by uploaded metadata. One value can be selected per column, and selections across columns are intersected with AND—for example, `gender = Female AND country = Algeria`. The interface reports the matching cohort size immediately.

For a filtered comparison, usable sentence scores are re-centered and re-scaled within the selected entities. The same subset is used for percentiles, distributions, entity correlations, and group-profile affinities. A group affinity correlates the submitted entity's sentence-level response vector with the selected cohort's average vector for that group; it describes model-behavior resemblance, not actual group membership. At least three reference entities are required. Metadata columns with 2–250 repeated values are available as filters; the stricter 2–8 limit applies to ellipse-plot groupings.

## Run artifacts

Each audit is persistent in `bias_workbench/runs/<project-id>/`:

```text
project.json                 task design and run state
generated_sentences.csv     labels, keywords, English sentences, translations
locales.json                translated prompts and few-shot examples
assisted_entities.csv       last AI-generated entity draft, before user edits
entities.csv                uploaded reference cohort
entity_scores.csv           one normalized score per entity and setting
report.json                 data used by the interactive report
contexts/*.npz              sentence-level reference statistics and vectors
additional_entities/*.json  saved additional-entity comparisons
```

The latest category-token preflight, including matched and observed tokens, is retained inside `project.json`.

The generated sentences and entity scores can also be downloaded from the website. Runs remain on disk after the web server stops; delete a run directory manually when it is no longer needed.

## Scope

This application is intended for local demonstrations and task prototyping. It does not provide authentication, multi-user isolation, rate limiting, or prompt-injection defenses, and should not be exposed directly to the public internet.
